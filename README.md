### Commit 1 Reflection Notes
---
Di Milestone 1, saya belajar membangun sebuah *web server single-threaded* sederhana menggunakan Rust. Langkah pertama yang dilakukan adalah membuat *server* mendengarkan koneksi TCP pada *socket* lokal di alamat `127.0.0.1:7878` melalui komponen `TcpListener`. Selanjutnya, mengimplementasikan `handle_connection` untuk menerima dan memproses *stream* data dari setiap koneksi TCP yang berhasil tersambung. Proses membaca permintaan HTTP dari *browser* ini dioptimalkan  `BufReader` agar arus data dapat dibaca secara terstruktur baris demi baris menggunakan `.lines()`. *Raw data* yang masuk tersebut kemudian diambil dan dikumpulkan ke dalam sebuah *Vector* hingga program mendeteksi adanya baris kosong penanda akhir permintaan. Melalui tahapan ini, saya dapat melihat langsung struktur komunikasi yang dikirimkan oleh *client*, seperti metode `GET` dan rincian *header* pendukung lainnya, sesaat sebelum *server* memberikan balasan.

### Commit 2 Reflection Notes
---
![Commit 2 Screen Capture](/assets/images/Commit2.png)

Di Milestone 2, saya mempelajari cara menambah fungsionalitas *web server* agar dapat merespons *request client* dengan mengirimkan sebuah berkas HTML. Langkah ini dilakukan dengan memodifikasi metode `handle_connection` untuk membaca isi `hello.html` menggunakan fungsi `fs::read_to_string`. Setelah berhasil dibaca, saya belajar membuat respons HTTP yang diawali dengan baris status `HTTP/1.1 200 OK` sebagai penanda bahwa permintaan berhasil diproses. Respons tersebut kemudian diberikan *header* `Content-Length` berdasarkan jumlah karakter teks, diikuti oleh isi kode HTML itu sendiri di bagian bawah. Seluruh rangkaian string respons itu kemudian dikonversi menjadi format *byte* agar dapat dikirimkan kembali ke *browser* menggunakan metode `stream.write_all`. Tahapan ini memberikan pemahaman yang kuat bagi saya tentang bagaimana sebuah *server* membuat struktur *response* sehingga halaman *web* buatan sendiri dapat dirender dengan baik di sisi *client*.

### Commit 3 Reflection Notes
---
![Commit 3 Screen Capture](/assets/images/Commit3.png)

Di Milestone 3, saya belajar bagaimana membuat *server* memvalidasi setiap permintaan yang masuk secara selektif. Sebelumnya, *server* selalu mengembalikan halaman `hello.html` secara utuh terlepas dari *endpoint* apa pun yang diminta oleh pengguna. Untuk memperbaiki keterbatasan tersebut, saya memodifikasi logika dengan menerapkan *pattern matching* pada baris pertama permintaan HTTP. Jika *browser* secara spesifik meminta rute utama (`GET / HTTP/1.1`), *server* akan membalas dengan status `HTTP/1.1 200 OK` dan berkas halaman `hello.html`. Sebaliknya, jika klien mencoba mengakses rute lain yang salah atau tidak terdaftar, *server* akan menangkapnya dan merespons menggunakan status `HTTP/1.1 404 NOT FOUND` beserta halaman `404.html`. Di tahapan ini saya juga belajar pentingnya melakukan *refactoring* kode pemrosesan agar logika percabangan respons menjadi jauh lebih efisien dan tidak ada dari *duplicate code*.

### Commit 4 Reflection Notes
---
Di Milestone 4, saya menyadari kelemahan dari arsitektur *web server single-threaded* yang telah dibangun sebelumnya. Untuk mengamati batasan ini secara langsung, saya menambahkan simulasi penundaan waktu dengan membuat rute baru `/sleep` yang menggunakan fungsi `thread::sleep` selama 10 detik. Saat saya mencoba mengakses rute `/sleep` di satu jendela *browser* dan rute utama `/` di jendela lainnya secara bersamaan, saya melihat secara bagaimana *server* tertahan. Halaman utama baru bisa dimuat dan dirender oleh *browser* setelah proses *sleep* selama 10 detik pada rute pertama selesai dieksekusi secara keseluruhan. Fenomena antrean yang saling menghalangi (*blocking*) ini membuktikan bahwa *server single-thread* tidak efisien dan bermasalah jika banyak pengguna mencoba mengaksesnya secara bersamaan. Hal ini memberikan motivasi yang sangat bagi saya untuk segera beralih menggunakan arsitektur *multithreading* pada tahap selanjutnya.

### Commit 5 Reflection Notes
---
Di Milestone 5, saya belajar mengatasi masalah *blocking* yang terjadi sebelumnya dengan meningkatkan arsitektur *web server* menjadi *multithreaded*. Untuk mewujudkan hal tersebut, saya mengimplementasikan struktur `ThreadPool` yang berfungsi untuk membatasi sekaligus mengelola jumlah *thread* pekerja secara efisien. Dibandingkan dengan membuat *thread* baru tanpa batas untuk setiap koneksi yang masuk, *server* kini mendistribusikan tugas penanganan tersebut ke dalam antrean yang akan dieksekusi secara bergantian oleh kumpulan *thread* yang tersedia. Pendekatan ini memecahkan masalah penundaan waktu tunggu, karena ketika ada permintaan rute yang memakan waktu lama, *thread* pekerja lain tetap siap melayani permintaan klien berikutnya secara instan tanpa hambatan. Selama pembuatan ini, saya juga mempraktikkan langsung penerapan konsep *Concurrency* di bahasa Rust, khususnya dalam mendesain sistem komunikasi dan berbagi *state* yang aman antar-*thread*. Pada akhirnya, tahapan ini menyempurnakan pemahaman saya dalam mendesain infrastruktur kode yang lebih optimal dan siap menghadapi beban konkurensi secara nyata.
